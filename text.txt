Functional Requirements:

The end flow of Movie ticket Booking system:

    user -> search the movie(related to location) -> movies details can shown -> select the theater -> 
    select the screen -> select the seat -> payment service -> confirmed payment -> ticket can send to user via email or sms .

Non-Functional Requirements:

Edges cases handles:
    what happened if the more than one user can access the same seat?

        case 1 -> i used an distributed cache for store the seat temporarily for the user the booking can confirmed within 
        the time window means after stored the seats in booking_seats DB and removed the lock from redis cache.
        i used an atomic operation in redis cache which user come first the redis can allocate for this user (using NX);
        
        case 2 -> simultaneously the key is insert in seat lock table temporarily and also prevent the inconsistency while 
        redis crashed.once the payment is succeed iam stored it in DB;

        it will check the redis in memory if key exists or not -> if it exits ,the response is already booked otherwise it moves 
        seat lock table and DB for confirmation is the key exists or not -> if it not exits it stored it an DB after payment 
        is succeed.

        
oAuth and JWT:

only use JWT(JSON Web Token):
    if the user signup using bcryptjs for password hashing
    after the user login it moves to backend server it creates a JWT use of secret key once it created it send back to client.
    the user before access the service the service can verify the user using new signature created using secret key and it matched with old 
    signature and proceed 

use both JWT and oAuth:
    but there is an drawback of using only JWT because if the JWT can stolen by attacker means it cause an problem so reduce this we use an 
    oAuth access token and refresh token. by using this access token is valid only 15 min and refresh token is valid 10 days. the access token 
    is expired the user send the refresh token to get the access token and again it works fine

only use oAuth(open Authorization):
    the user can login using another trusted provider or another service. it give an google token id generated using google private key using
    this token id, the backend can verify the user using google public key.


Edges cases for payment:
    If the user click the payment button twice?
        the user click the payment button twice the DB insert duplicate key also...to overcome this we store an unique id(idempotency key).
        this is unique for each booking.
        Before the user click the payment or initiate the payment...the booking id is generated and it should be unique so it act as an 
        idempotency key here. if the user click more than couple of time the DB stores only one request which one comes first it will store and 
        next insert should be fail and return already paid or key exits or in processing state.
        if the DB have an key already means it stop the payment..this ensure the double ticket and double payment is not happening


to run on the other system :
Prerequisites: Only Docker Desktop installed
# Step 1: Clone the repository
git clone https://github.com/2005arun/Distributed_Movie_Booking_System.git
cd Distributed_Movie_Booking_System

# Step 2: Start everything with one command
docker compose -f docker-compose.prod.yml up -d

# Wait 30 seconds for all containers to initialize

# Step 3: Access the app
# Frontend:    http://localhost:5173
# API Gateway: http://localhost:3000
# RabbitMQ:    http://localhost:15672 (guest/guest)

our system :
# Step 1: Kill all old processes and containers
taskkill /F /IM node.exe 2>$null
docker stop $(docker ps -q) 2>$null
docker rm $(docker ps -aq) 2>$null

# Step 2: Start PostgreSQL & RabbitMQ only (in Docker)
cd D:\booking_movie_ticket
docker compose up -d postgres rabbitmq

# Wait 15 seconds for database to initialize

# Step 3: Install dependencies (if first time)
npm install
cd gateway && npm install && cd ..
cd services/auth-service && npm install && cd ../..
cd services/movie-service && npm install && cd ../..
cd services/show-service && npm install && cd ../..
cd services/booking-service && npm install && cd ../..
cd services/payment-service && npm install && cd ../..
cd services/ticket-service && npm install && cd ../..
cd services/notification-service && npm install && cd ../..
cd frontend && npm install && cd ..

# Step 4: Start all 8 backend microservices
node start-all.js

# Step 5: In a NEW terminal, start frontend
cd D:\booking_movie_ticket\frontend
npm run dev

To Stop Everything Later:
taskkill /F /IM node.exe 2>$null
docker stop mtbs-rabbitmq
